App Store Screenshot Playbook

Screenshot generation system using claude-skill-aso-appstore-screenshots. Standalone process — run when any app needs App Store listing screenshots.

How to Use This Playbook

Input: An app ready for App Store listing (new launch or existing app needing screenshot refresh)

Process: Walk through Steps 1→7 sequentially. Each step builds on the previous.

Output: 5 App Store-ready screenshots (1290×2796px) + showcase preview image

Time: ~1-2 hours total (benefit discovery + screenshot capture + generation)

Tool Required: claude-skill-aso-appstore-screenshots
Install: claude install-skill github.com/adamlyttleapps/claude-skill-aso-appstore-screenshots
Dependencies: Python 3 + Pillow, SF Pro Display Black font, Gemini MCP server

1. When to Run

This playbook is a standalone process — not tied to any sprint or development workflow. Run it whenever:

TriggerWhenGoal
NEW LAUNCH App is built, ready for App Store submission Create initial screenshot set for listing
ASO REFRESH Existing app needs better conversion Replace screenshots to improve download rate

2. Core Rules

Non-Negotiable Rules

  • 5 screenshots per app — maps 1:1 to 3-5 benefits from discovery
  • Core loop screens only — skip settings, onboarding, paywall screens
  • Staged ideal data — populate with impressive-looking fake data (big numbers, full history, nice charts)
  • Cleanest version wins — when picking from AI-generated versions, choose most readable, least noisy
  • Auto-detect brand color — use color from app's codebase design, don't invent

3. Step 1 — Benefit Discovery MOST CRITICAL

This phase sets the foundation for everything. Follow the 4-step process strictly. Do not rush. Do not skip to screenshot generation.

Step 1.1: Analyze the Codebase

Read the app's codebase thoroughly. Build a mental model of:

  • What the app does — UI files, screens, components. What can the user actually DO?
  • Who it's for — models, data structures. What domain does it operate in?
  • What's premium — IAP, subscriptions, feature flags. What's the paid offering?
  • What it highlights first — onboarding flows, README, marketing copy in code

Step 1.2: Ask Clarifying Questions

After analysis, present what you've learned and ask targeted questions to fill gaps:

  • "Based on the code, this appears to be [X]. Is that right?"
  • "Who is your target audience? (age, interests, skill level)"
  • "What's the #1 reason someone downloads this app?"
  • "Who are your main competitors?"
  • "What do your best reviews say? What do users love most?"

Don't ask questions the code already answers.

Step 1.3: Draft Core Benefits STRICT RULES

Draft 3-5 core benefits. Each benefit MUST:

  1. Lead with an action verb — TRACK, CRUSH, BUILD, FIND, SAVE, LEARN, SEE, SPEAK, MASTER, HEAR...
  2. Focus on what the USER gets, not what the app does technically
  3. Be specific enough to compel download — vague = scroll past
  4. Answer the user's unspoken question: "Why should I download this instead of scrolling past?"

Benefit Examples — 3 App Types

App Type BAD (generic) GOOD (specific)
Debt Payoff Tracker MANAGE YOUR FINANCES CRUSH DEBT FASTER
VIEW YOUR DEBTS SEE YOUR DEBT-FREE DATE
TRACK PAYMENTS SAVE THOUSANDS IN INTEREST
Habit Tracker BUILD BETTER HABITS TRACK YOUR CLEAN STREAK
SET DAILY REMINDERS NEVER BREAK THE CHAIN
VIEW YOUR PROGRESS SEE HOW FAR YOU'VE COME
Language Learning LEARN A NEW LANGUAGE SPEAK SPANISH IN 10 MINUTES A DAY
STUDY VOCABULARY MASTER 1000 ESSENTIAL WORDS
PRACTICE DAILY HEAR YOUR PRONUNCIATION SCORE

What Makes a BAD Benefit

  • Generic verbs: MANAGE, USE, ACCESS, GET
  • Describes the app, not the user's gain
  • Could apply to any app in the category
  • No emotional pull — user scrolls past

What Makes a GOOD Benefit

  • Strong action verbs: CRUSH, MASTER, TRACK, SEE
  • Describes what the user GETS or FEELS
  • Specific to THIS app's unique loop
  • Creates urgency or curiosity — user taps to download

Step 1.4: Refine with User

DO NOT proceed until user explicitly confirms the benefits. This is iterative:

  • Let user reorder, reword, add, or remove benefits
  • Suggest alternatives if user isn't happy
  • Explain reasoning — why a particular verb or phrasing converts better
  • User has final say, but push back (politely) on generic choices

4. Step 2 — Simulator Screenshots

Capture 5-8 simulator screenshots to feed the generation tool. Quality in = quality out.

What to Capture

CAPTURE — Core Loop Screens

  • Main dashboard / home screen
  • Primary action screen (the thing the user DOES)
  • Result / reward screen (the thing the user GETS)
  • History / progress screen
  • Different data states of key screens (populated, mid-action)

SKIP — Non-Core Screens

  • Settings / preferences
  • Onboarding / tutorial
  • Paywall / subscription
  • Login / registration
  • Empty states
  • Debug or developer screens

Data in Screenshots

Stage ideal-looking data. Screenshots should look impressive:

  • Big numbers, full history, completed streaks
  • Nice-looking charts with upward trends
  • Multiple items/entries showing the app is well-used
  • Consistent light OR dark mode across all screenshots
  • Clean status bar (no low battery, no debug indicators)

Never show: placeholder data, empty lists, "lorem ipsum", sparse content, error states

Minimum Screenshot Count

App ScreensStrategyScreenshot Count
5+ core screens One screenshot per screen 5-8
3-4 core screens Key screens + different data states for variety 5-6
1-2 core screens Multiple data states + interaction moments 5 (minimum)

5. Step 3 — Localization Check

Not all apps need localized screenshots. Check per app:

LOCALIZATION CHECK: Does this app target a non-English market? YES / NO If YES: - Target language(s): ___ - Translate benefit headlines: YES - Different screenshots needed? (cultural data, RTL layout): YES / NO - If different screenshots: retake simulator screenshots with localized data If NO: - English only. Proceed to Step 4.
When to localize: Apps targeting specific regions (Arabic markets, Latin America, South Africa, etc.) benefit from localized headlines. Same screenshots, translated text is the minimum. Full localization (different data, cultural cues) is ideal but optional.

6. Step 4 — Brand Color

Auto-detect from app's codebase design color. The tool analyzes your app's theme/design system and picks the primary brand color automatically.

How It Works

  • Tool reads your codebase for color constants, theme files, design tokens
  • Selects the dominant brand color
  • Uses it as the screenshot background color
  • All 5 screenshots share the same background for consistency

Override

If the auto-detected color looks wrong (too light, too muted, poor contrast with white text), you can override with a specific hex value. But prefer the auto-detected color — it matches your app.

7. Step 5 — Generation

The tool handles this automatically via its 2-stage pipeline. You just trigger it.

Pipeline

1
Scaffold (Deterministic) — compose.py creates pixel-perfect 1290×2796 PNG with benefit headline text + device frame + your simulator screenshot composited inside. Same layout every time.
2
AI Enhancement (Gemini/Nano Banana Pro) — Replaces device frame with photorealistic mockup, adds optional breakout elements, polishes shadows and depth. Generates 3 versions per screenshot.
3
Crop & Resize — sips crops from 9:16 to Apple's exact aspect ratio, resizes to 1290×2796px. App Store-ready.

8. Step 6 — Version Selection

Rule: Pick the cleanest version. You're not a designer. Don't overthink it. Use this checklist:

Selection Checklist

CriteriaCheck
Headline text is fully readable No text cut off, no overlapping elements
Device frame looks clean No warped edges, no missing corners, no artifacts
App screenshot is visible and sharp Not blurry, not obscured by breakout elements
No AI artifacts No weird shadows, no distorted UI elements, no hallucinated buttons
Consistent with other screenshots in the set Same device frame style, same text treatment, same overall feel

PICK This Version

  • Text is crisp and centered
  • Device frame is clean and realistic
  • Screenshot inside device is clear
  • Minimal visual noise
  • Breakout element (if any) enhances, doesn't distract

REJECT This Version

  • Text is partially cut off or hard to read
  • Device frame has warped or blurry edges
  • AI added weird visual elements that don't exist in app
  • Breakout element covers important UI
  • Looks different from other screenshots in set
If all 3 versions have issues: Regenerate. The tool can produce 3 new versions. Check if the scaffold itself has problems (bad screenshot crop, text too long) — fix input before regenerating.

Upload Order

Screenshots appear in App Store in the order you upload. Order matters — first screenshot gets the most views.

PositionWhat to ShowWhy
1st Your strongest benefit — the #1 reason to download 80% of users only see the first 1-2 screenshots
2nd The core loop in action — what using the app looks like Shows the app is real and functional
3rd Secondary benefit — reinforces value For users who swipe past the first two
4th-5th Supporting benefits — additional features For detail-oriented users comparing apps
