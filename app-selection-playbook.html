App Selection Playbook

Sprint Decision System — Pick the right app to build next from validated ideas

How to Use

  • Input: Ideas that passed Astro MCP scoring (>50) AND Feature Validation (BUILD verdict)
  • Process: Walk through Steps 1→8. Each step filters or ranks.
  • Output: Ranked top 3 with per-idea breakdown → Sprint #1

Core Philosophy

  • Always copy proven revenue apps. Never invent.
  • Differentiate through fixing verified user pain only.
  • Missing features in competitors = users don't need it. Never add it.
  • AI must never flatter. Question every "advantage" claim.
  • No designer — copy UI from competitors (one primary source).

Constraints

  • Money first priority
  • Sprint mode: 2-3 weeks to ship
  • Ship = core loop + paywall only (Option A from feature validation)
  • Post-launch kill line: $300/mo within 60 days

Step 1: Revenue Verification

  • Confirm competitors make real money
  • Source: SensorTower, data.ai, AppMagic, Astro MCP, or manual estimation (price × review volume)
  • KILL if no competitor shows meaningful revenue

AI Prompt Template

Check revenue data for [app category]. Sources: SensorTower, data.ai, AppMagic, App Store rankings, or estimate from subscription price x review count. List top 5 competitors with estimated revenue. KILL if no competitor shows >$10K/year revenue.

Step 2: Competitor Review Analysis (pre-sprint)

  • AI-assisted analysis
  • Read top 3 competitors × 50 recent negative reviews each
  • Categorize complaints with frequency counts
  • Output: ranked pain point list

AI Prompt Template

For [competitor app name], scrape/export the 50 most recent 1-star and 2-star reviews. Categorize complaints into themes. Count frequency per theme. Output: ranked list of pain points by frequency. Identify max 1 pain-point feature that 50+ reviews demand.

Step 3: Anti-Flattery Critical Thinking Check

CRITICAL — The Rule

Before claiming ANY advantage over a competitor, pass this checklist:

Check Question Default Answer
□ Why doesn't the market leader have this feature? They tested and removed it, or users don't need it
□ Why does the competitor do it THIS way? They have a reason — technical, monetization, or user research
□ Is this "advantage" from real user reviews or AI flattery? If you can't cite 50+ reviews, it's flattery
□ Can I point to 50+ reviews demanding this specific thing? If no, DON'T add it

Anti-patterns

WRONG

"App A doesn't have dark mode — great opportunity for us!"

RIGHT

"App A doesn't have dark mode. They have 500K users without it. Users don't need it. Skip."

WRONG

"App B's onboarding is 6 screens — we can do better with 3!"

RIGHT

"App B uses 6 onboarding screens. Why? Maybe they tested shorter and conversion dropped. Copy their flow unless reviews specifically complain about onboarding length."

Step 4: UI Reference Selection

  • Pick ONE app as primary UI source (usually prettiest / highest-rated competitor)
  • Copy 80% of layout from primary source
  • Fill gaps from other competitors
  • No original design — copy directly

AI Prompt Template

For [app category], identify the best-looking competitor app. Screenshot every screen of that app. This is our primary UI reference -- copy layout, spacing, typography patterns. For screens the primary app doesn't have, find UI reference from secondary competitors.

Step 5: Personal Itch Filter

Must pass 2-of-3:

Test Question Pass Criteria
Recency Have I experienced this pain in the last 7 days? Yes = pass
Behavior Have I tried solving this with a workaround? (spreadsheet, other app, notebook) Yes = pass
Rant Can I rant about this problem for 2 minutes without stopping? Yes = pass
Pass 2+ = proceed  •  Pass 0-1 = skip this idea (not a real itch)

AI Prompt Template

For idea [name], answer honestly: 1. Recency: When did I last feel this pain? (must be within 7 days) 2. Behavior: What workaround am I currently using? (must name a specific tool/method) 3. Rant: What specifically frustrates me about this problem? (must sustain 2 minutes) Score: ___/3. Must be >=2 to proceed.

Step 6: Rank by Speed-to-Ship

  • Count screens needed for core loop + paywall
  • Estimate data model complexity (entities, relationships)
  • Fastest to ship = highest rank

Step 7: Tiebreaker — Monetization Signal

  • If two ideas have same speed rank, pick the one where competitors make MORE money
  • Compare competitor subscription prices and estimated revenue
  • Higher proven revenue market wins

Step 8: Optional Accelerators

Two optional bumps that can change ranking:

Timing Accelerator

  • Seasonal demand (Ramadan, tax season, back-to-school)
  • Trending topic or cultural moment
  • Platform featuring a category
  • If tailwind exists NOW → bump up

Tech Overlap Bonus

  • Can reuse code/infra from existing apps
  • Shared payment flow, onboarding pattern, component library
  • Reusable = faster to ship = bump up

Post-Sprint Rules

  • App hits $300/mo in 60 days → double down, keep building
  • App fails $300/mo gate → grab #2 immediately, no re-evaluation
  • Sprint cycle complete → re-evaluate ALL survivors with full Playbook 3
  • Non-picked survivors stay in pool, re-evaluated each sprint
