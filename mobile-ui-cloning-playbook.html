Mobile App UI Cloning Playbook

Screenshot-to-Code system for React Native / Expo using Claude Code. Clone competitor UI, validate against user stories, eliminate the "AI look."

How to Use This Playbook

Input: PRD / feature list + competitor app screenshots (App A, B, C) + primary UI reference app

Process: 3 phases — Prep → Per-Screen Generation → Flow Validation

Output: Full React Native UI codebase, validated against user stories, visually faithful to competitor apps

Tool: Claude Code (screenshot input → React Native code output, no extra tools)

Stack: React Native / Expo, raw components + StyleSheet (no component library)

1. The Problem

When building mobile app UI with AI, three things go wrong:

Problem 1: Lost Context

You have pieces in your head — "this part from App A, this from App B, this from App C." But you manually prompt AI to connect them, lose context between sessions, and forget decisions.

Problem 2: Looks Like AI

AI generates UI that looks generic — default icons, even spacing, safe color palettes, flat cards. It doesn't look like a real app. It looks like a template.

Problem 3: No Validation Loop

AI generates pretty screens that don't serve the actual user flow. No connection between the UI and the tasks the user needs to complete. Screens look nice but don't work as a product.

The Solution

This playbook solves all three by:

  • Context: Standard design token template + screenshot-to-story mapping — nothing stays in your head
  • Anti-AI: Extract exact design tokens from competitor screenshots — AI is constrained, not creative
  • Validation: Every screen is generated from a user story and validated against it — AI checks first, you approve

2. Anti-"AI Look" Rules CRITICAL

These 5 rules prevent AI from generating generic-looking UI. Apply to EVERY screen.

Problem AI Default (BAD) Cloned (GOOD) Fix
1. Default Icons Uses generic icon names like "home", "settings", "user" Uses exact icon names/style from competitor Identify competitor's icon library (SF Symbols, Feather, Phosphor, custom). Specify exact icon names in prompt.
2. Wrong Spacing Even 16px padding everywhere Tight/loose rhythm matching real app Extract exact padding/margin values from competitor screenshot. Measure gaps between elements.
3. Missing Micro-Details Flat cards, no shadows, uniform border radius Subtle shadows, varied radius, gradients Extract shadow values, border radius per component type, gradient colors from competitor.
4. Generic Colors Safe blues (#007AFF), grays (#666) Exact hex values from competitor's palette Color-pick from competitor screenshots. Extract primary, secondary, background, text, accent colors.
5. Wrong Components Material Design defaults, basic TextInput Custom-styled components matching competitor Use raw React Native + StyleSheet. No component library. Style every component to match screenshot.
The Rule: AI should be a copier, not a designer. Every visual decision comes from a competitor screenshot — never from AI's "taste."

Per-Screen Overrides

When borrowing a screen from App B or C (not the primary reference), extract only what's different:

// Screen: ActivityDetail
// Borrowed from: App B
// Overrides from primary design system:

const screenOverrides = {
  cardBorderRadius: 20,    // App B uses rounder cards
  listItemPadding: 18,     // App B has more breathing room
  // All other tokens: use design-tokens.ts defaults
};

Only override what's visually different. Keep everything else from the primary design system for consistency.

4. Phase 1 — Prep

Before generating any UI code, complete all 4 prep steps. Skipping any step = generic output.

Step 1: Generate User Stories from PRD MANDATORY

You always skip this step. That's why AI generates screens that don't connect to real user flows. This step is now mandatory.

AI generates medium-detail user stories from your PRD/feature list. You confirm.

User Story Format (Medium Detail)

STORY: [Story name] AS A: [user type] I WANT TO: [action — specific tap/input] SO THAT: [outcome — what user gets] FLOW: 1. User sees [screen name] with [key elements visible] 2. User taps [specific element] 3. Screen shows [response — what changes] 4. User [next action] 5. Result: [end state] REQUIRED ELEMENTS: - [button/input/display element 1] - [button/input/display element 2] - [button/input/display element 3]

Example: Debt Payoff Tracker

TOO SIMPLE (not enough to validate)

User logs a payment.

MEDIUM DETAIL (just right)

STORY: Log Debt Payment
AS A: debt holder
I WANT TO: log a payment I made
SO THAT: I see my debt-free date move closer

FLOW:
1. User sees debt list with each debt showing
   name, balance, minimum payment
2. User taps a specific debt card
3. Screen shows debt detail with "Log Payment" button
4. User taps "Log Payment" → input modal appears
5. User enters amount → taps "Save"
6. Debt balance updates, debt-free date recalculates
7. Result: updated balance + new debt-free date visible

REQUIRED ELEMENTS:
- Debt card with name, balance, min payment
- "Log Payment" button
- Amount input with number keyboard
- Updated debt-free date display
AI PROMPT — Step 1: Read my PRD at [path]. Generate user stories for each core feature. Use medium detail: AS A / I WANT TO / SO THAT + numbered flow + required elements. Do NOT add features not in the PRD. Do NOT make up user flows — derive from the PRD's feature descriptions. Output all stories. I will confirm or adjust before proceeding.

Step 2: Confirm User Stories

Review each story. Check:

  • Does the flow match what you want the app to do?
  • Are all required elements listed?
  • Is anything missing from the PRD?
  • Is anything added that's NOT in the PRD?

Do NOT proceed until all stories are confirmed.

Take screenshots of your primary UI reference app (chosen in App Selection Playbook). Feed them to Claude Code to fill the design-tokens.ts template.

5. Phase 2 — Per-Screen Generation

Generate one screen at a time. Each screen goes through: Generate → Render → AI Validate → You Approve.

Per-Screen Loop

1
Feed AI: Screenshot + Story + Design Tokens
Give Claude Code three inputs simultaneously: the competitor screenshot to clone, the user story this screen must serve, and the design-tokens.ts file (+ any overrides for this screen).
2
AI Generates React Native Code
Output: raw React Native components + StyleSheet. No component library. All style values from design-tokens.ts. Layout matches the competitor screenshot. Interactive elements match the user story's required elements.
3
Render in Simulator
Run the app in iOS Simulator. Take a screenshot of the rendered screen.
4
AI Validates Against User Story (First Pass)
Feed the rendered screenshot back to Claude Code with the user story. AI checks: are all required elements present? Are they interactive? Does the layout support the flow?
5
You Validate (Final Approval)
You check: does it look like the competitor screenshot? Does spacing feel right? Any "AI look" issues? Would you download an app that looks like this?
6
Pass or Fix
If pass → move to next screen.
If fail → identify what's wrong, fix, re-render, back to step 4.

AI Generation Prompt

AI PROMPT — Per-Screen Generation: SCREENSHOT: [attached competitor screenshot] Clone this screen's visual layout exactly. USER STORY: [paste the full user story for this screen] DESIGN TOKENS: Use values from design-tokens.ts [paste any per-screen overrides if applicable] RULES: - React Native + Expo, raw components + StyleSheet only - NO component libraries (no Paper, no NativeBase, no Tamagui) - ALL colors from design-tokens.ts — do NOT pick your own colors - ALL spacing from design-tokens.ts — do NOT use even spacing - ALL border radius from design-tokens.ts — do NOT use uniform radius - Match icon style from design-tokens.ts icons.library - Every element in REQUIRED ELEMENTS must be present and interactive - Layout must match the screenshot — same proportions, same hierarchy - Include shadows as specified in design-tokens.ts OUTPUT: Complete screen component file with StyleSheet.

AI Validation Prompt

AI PROMPT — Validation: RENDERED SCREENSHOT: [attached screenshot of rendered screen] COMPETITOR SCREENSHOT: [attached original competitor screenshot] USER STORY: [paste user story] CHECK: 1. Are ALL required elements from the story present on screen? 2. Is each required element visually accessible (not hidden, not tiny)? 3. Does the layout match the competitor screenshot? 4. Are colors matching design-tokens.ts? (no AI-default blues/grays) 5. Is spacing varied (not uniform 16px everywhere)? 6. Are shadows/border-radius matching tokens? 7. Would a user be able to complete the story flow on this screen? OUTPUT: - PASS or FAIL - If FAIL: list specific issues with line-level fix suggestions

6. Phase 3 — Flow Validation

After all screens pass individually, test the complete flow end-to-end.

Flow Validation Checklist

Check Question If Fail
Navigation Does every screen connect to the next correctly? Fix navigation wiring, check stack/tab setup
Story Completion Can user complete every user story start-to-finish? Identify missing step, add screen or element
Back Navigation Can user go back from every screen? Add back button/gesture handler
Data Flow Does data entered on one screen appear correctly on another? Fix state management / props passing
Visual Consistency Do all screens look like they belong to the same app? Check design-tokens.ts usage, fix outlier screens
No Dead Ends Is there any screen the user can reach but can't leave? Add navigation escape (back, close, dismiss)
AI PROMPT — Flow Validation: I've built all screens. Here are screenshots of every screen in the app: [attach all rendered screenshots] Here are all user stories: [paste all stories] CHECK: 1. Can every user story be completed start-to-finish? 2. Are all screen transitions logical? 3. Is any required element missing from any screen? 4. Do all screens share consistent visual style? 5. Are there any dead ends? OUTPUT: - PASS or list of issues with specific screen + fix suggestions
