Feature Validation Playbook

5-step gate to validate any new app feature idea before building. Run each step sequentially. Steps 1-2 are hard kill gates — fail = kill the idea. Steps 3-5 are soft gates — fail = reshape, not kill.

How to Use This Playbook

Input: One feature idea (name + rough description)

Process: Walk through Steps 1→5 sequentially. Fill each section. Apply kill/reshape criteria at each gate.

Output: Filled template with per-step verdict + final decision: BUILD or RESHAPE or KILL

AI INSTRUCTION: When user provides a feature idea, walk through all 5 steps below. Fill each section with concrete answers. Apply gate criteria strictly. Output a completed template with verdict per step and final decision at bottom. Be concise. Sacrifice grammar for clarity. No filler.

Validation Overview

Step Name Gate Type Kill Criteria
1 Core Value HARD KILL Can't state value in one sentence → kill idea
2 Core Loop HARD KILL Can't name action → immediate reward → kill idea
3 Accessory Features SOFT — RESHAPE Remove accessory, loop still works → cut it
4 Surface Area Check SOFT — RESHAPE Exceeds 5-7 screens or 3 elements/screen → simplify
5 Retention Hook SOFT — RESHAPE Can't name specific mechanism → redesign

Step 1: Core Value HARD KILL

The Rule

Define the core value of this feature in one sentence. One feature at a time. If you can't — the idea is too vague or too broad. Kill it.

Gate Question

"What single problem does this feature solve, and for whom?"

Pass Criteria

  • One sentence, no conjunctions ("and", "also", "plus")
  • Names a specific user pain or desire
  • A stranger could read it and understand

Examples

PASS
"Users forget to drink water throughout the day — this feature reminds them and tracks intake."

"Debt holders can't see which debt to pay first — this ranks debts by cost-saving impact."
FAIL
"A comprehensive health dashboard with hydration, sleep, exercise, and nutrition tracking."
→ Too many things. Which ONE?

"Make the app better and more engaging."
→ No specific problem named.

Step 2: Core Loop HARD KILL

The Rule

Map Action → Immediate Reward. The reward must happen in the same session, within seconds of the action. If the reward is delayed (next day, end of week, after 30 entries) — the loop won't drive daily usage. Kill it.

Gate Questions

  • "What does the user DO?" (the action — must be a specific tap/input)
  • "What do they GET back immediately?" (the reward — must be visible/felt within seconds)
  • "What type of reward?" (categorize it)

Reward Types

Type Definition Example
Intrinsic User feels progress, relief, or satisfaction Log payment → debt bar shrinks visually
Data User gets insight they didn't have before Input expense → "you're 20% under budget"
Social Recognition from others Complete challenge → badge visible to friends
Unlockable Access to something new Finish lesson → next level unlocks

Examples

PASS
Action: Tap "drank water"
Reward: Hydration bar fills up instantly (intrinsic)
Delay: 0 seconds

Action: Log a debt payment
Reward: Debt-free date moves closer, shown immediately (data + intrinsic)
Delay: 0 seconds
FAIL
Action: Input daily expenses
Reward: Monthly spending report
Delay: 30 days → dead loop

Action: Complete onboarding quiz
Reward: "We'll send personalized tips via email"
Delay: hours/days → no immediate payoff

Step 3: Accessory Features SOFT — RESHAPE

The Rule

Add only what supports the core loop. For every proposed accessory, apply the litmus test:

"If I remove this accessory, does the core loop BREAK?" → Yes, it breaks → KEEP (load-bearing) → No, loop still works → CUT for v1 (decorative)

Common Traps

Accessory Feels Like Actually Is Verdict
Push reminders to trigger action Nice to have Drives the loop — triggers action KEEP
Share stats to social media Supports engagement Vanity feature, loop works without it CUT
Track a second related metric Natural extension Second core loop disguised as accessory CUT
History/log of past actions Basic feature Reinforces reward visibility KEEP
Customizable themes/colors Polish Zero loop impact CUT

Step 4: Surface Area Check SOFT — RESHAPE

The Rule

The entire feature (core loop + kept accessories) must fit within:

  • 5-7 new screens maximum
  • Max 3 new interactive elements per screen (buttons, inputs, toggles, pickers)

If it exceeds either limit → too complex. Split the feature or cut accessories.

Screen Budget Reference

Feature Shape Typical Screens
Simple CRUD (add/edit/delete/list) 3-4
Onboarding wizard 3-5
Dashboard + drill-down detail 2-3
Settings/config page 1-2
Payment/subscription flow 2-3

Complexity Signals

Under budget
5 screens, 2 elements each = 10 total interactions
Clean, focused, buildable in days
Over budget
9 screens, 5 elements each = 45 total interactions
Scope creep territory, split into phases

Step 5: Retention Hook SOFT — RESHAPE

The Rule

Name the specific mechanism that pulls users back to the app. Must be concrete, not "good UX" or "push notifications." Pick from known patterns:

Retention Mechanisms

Mechanism How It Works Example
Unfinished State Progress bar incomplete, streak alive, goal partially done — user feels pull to complete Duolingo streak, 70% hydration bar, "3/5 debts paid off"
Scheduled Reward Something valuable appears at a known future time — user returns to claim it "Daily insight unlocks at 9am", "Weekly debt report ready Monday"
Decaying Value Data or progress degrades if not maintained — user returns to keep it fresh Habit tracker marks "missed" days, streak resets, "last logged 3 days ago" warning
External Trigger Real-world event triggers app relevance — notification ties to user's actual life "Bill due in 2 days", "It's 2pm — you haven't logged water since 10am"

Gate Question

"Why would the user open this app again tomorrow — specifically because of this feature?"

If the answer is vague ("because it's useful") → no hook. Redesign.

Final Verdict

Decision Rules

Scenario Verdict
Steps 1-2 PASS + Steps 3-5 PASS BUILD — proceed to implementation
Steps 1-2 PASS + any of Steps 3-5 need work RESHAPE — core idea valid, adjust scope/hooks/surface
Step 1 or Step 2 FAIL KILL — idea fundamentally broken, don't build

Worked Example: Water Tracker — "Hydration Reminder"

Step 1 — Core Value PASS

Feature: Hydration Reminder

Core value: Users forget to drink water throughout the day — this tracks intake and shows progress visually.

Target user: Health-conscious adults who sit at desks all day.

Step 2 — Core Loop PASS

Action: Tap "drank water" button

Reward: Hydration bar fills up instantly with satisfying animation

Reward type: Intrinsic (visual progress)

Time to reward: <1 second

Step 3 — Accessories PASS

Reminder notificationsRemove → users forget to log → loop diesKEEP
History logRemove → no visible streak → reward weakensKEEP
Share to InstagramRemove → loop works fineCUT
Track caffeineRemove → loop works fine (it's a second loop)CUT

2 kept, 2 cut. Under threshold.

Step 4 — Surface Area PASS

  1. Home screen with hydration bar + log button (2 elements)
  2. History list view (1 element: scrollable list)
  3. Reminder settings (3 elements: toggle, time picker, frequency)
  4. Daily goal setting (2 elements: slider, save button)

4 screens, max 3 elements. Under budget.

Step 5 — Retention Hook PASS

Mechanism: Unfinished State + External Trigger

Implementation: Hydration bar resets daily at midnight. At 2pm if bar is under 50%, push notification: "You're at 40% — 3 glasses to go."

Why user returns: Bar is visibly incomplete. Notification ties to real-world need.

VERDICT: BUILD
